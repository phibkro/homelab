# Checkpoint — bootstrap (the generator)

Run this **once per project** (or to *refresh* a stale instance). It analyses the
codebase and writes `instances/<repo>.md` — the grounded config the run procedure
loads. Output is a *derivation* of the project: re-run it when the project's gate,
docs, or conventions move.

Lean on what the project already documents (its `CLAUDE.md`/`README`/`CONTRIBUTING`
usually state the verify command and the invariants) — **derive from those, don't
re-invent**. Single-source: the instance points at the project's own truth.

## 1. Discover the gate

Find the canonical "is it green" command and its **real-exit-code** form. Sources:
`justfile` / `Makefile` / `package.json` scripts / `CONTRIBUTING` / CI workflow /
`CLAUDE.md` "how to verify" / pre-commit hooks / lint+format config.

Record: the **verify gate** + its real-exit-code form, the **fast per-file loop**
(G0/G1), the **invariant guards** (hooks, audit/lint scripts, type-check), and any
**baseline** (distinguish a *tolerated-red* the gate ignores from a *deliberately-demoted*
check run off-gate — both mean "gate on new violations", differently).

Three shapes the gate can take — name which:
- **one command** (e.g. `lake build`) — simplest;
- **an orchestrator** (a wrapper script) — may need an interpreter that isn't provisioned,
  or may `exit(0)` regardless; record its *sub-checks* and how to read its *real* result
  (parse output if the exit code is meaningless);
- **split by grade into verify vs apply** — a safe verify gate (`nix flake check`, unit
  tests) AND a *mutating* apply/deploy step (`just rebuild`, `git push` to prod) that is
  **operator-gated and never auto-run**. Record the apply step as *identify-only*.

**Done when** you can name the verify gate (and its real result), the guards, and — if it
exists — the mutating apply step flagged never-auto-run.

## 2. Inventory the artifacts

Enumerate the derived-description files: top-level `*.md`, `README`, a
`docs/` tree, ADR directory, and any volatile-state doc (`CONTEXT`/`STATUS`/
`PATH`-like). Classify each by **Diátaxis mode → volatility** (`REFERENCE.md`):

- mark **generated** artifacts (drift-proof — they leave the survey),
- mark the **volatile-state** doc(s) (always in the blast radius),
- rank the rest Reference(HIGH) → How-to → Explanation/ADR → Tutorial(LOW).

**Done when** every doc-like file is classified, and you've flagged which are
generated vs hand-maintained.

## 3. Discover work-type conventions

**Discover the repo's actual vocabulary from `git log`** — do NOT impose
`feat`/`fix`/`chore`. Many repos use domain scopes (`kernel:`, `spec:`, `feat(worker)`,
`feat(networking)`), and the **scope frequently routes the survey more than the type**.
Also note: ADR discipline, a path/checkpoint/roadmap system, and changelog cadence
(per-commit vs batched-digest — the cadence sets the grain; absence is a finding too).

**Done when** you can map each *real* work-type/scope → the artifacts it tends to invalidate.

## 4. Discover idiosyncrasies

The project-specific invariants the checkpoint must guard, usually stated in
`CLAUDE.md` "invariants / do-not" sections, CI rules, or test conventions.
Examples: a kernel's primitive count + axiom hygiene + frozen-statement rule; a
web app's migration files + API contracts + generated clients; a library's
public-API + semver surface.

**Done when** the project's non-obvious "what must stay true" list is captured.

## 4b. Assess the surface & recommend climbs

Walk the nine maintenance surfaces (`REFERENCE.md` → "full maintenance surface":
build · test · docs · deps · cruft · debt · security · architecture · decisions).
For each, record (a) whether the project *has* it, and (b) which **rung** it sits
on — *generate* (drift impossible), *test* (CI catches it), or *survey* (hand-checked
hope). Then flag the **climbs**: a surface stuck on *survey* that a known tool could
pull to *test* is a maintenance-improvement recommendation, not just a status —
e.g. hand-checked import-layering → a fitness function; hand-surveyed Reference docs
→ generation; no secret-scan → `gitleaks`; prose tied to code → AST-anchors.

**Done when** every surface is rung-tagged and the climb opportunities are listed.

## 5. Emit the instance

Write `instances/<repo>.md` with: the gate command + guards (§1), the
artifact-volatility map (§2), the work-type→survey table (§3), the idiosyncrasies
(§4), and a one-line **worked example** from a recent real change. Header it
*"Generated by checkpoint/BOOTSTRAP — <date>; regenerate when gate/docs/conventions
move."*

**Done when** the instance exists and a fresh reader could run a checkpoint from
it alone.

## 6. Validate the instance

Run the **verify** gate once — but **only if it's safe, fast, and provisioned**.
If it mutates state (rebuilds a machine, deploys), needs credentials/network you lack,
or its interpreter isn't installed, **identify it, don't run it** (note "identified, not
run — mutates / heavy / unprovisioned"). Read its **real** result: if the gate is an
orchestrator that `exit(0)`s regardless, parsing its output is the *only* correct read,
**not** a fallback — do not reach for `$?`. Also pick out the **cheapest safe sub-check**
(an offline, zero-dep one — a doc/format/type check) and record it in the instance as the
recommended **G0/G1 fast gate**. Spot-check the artifact inventory against
`git ls-files '*.md' docs/` — no major doc missed.

**Done when** the verify gate ran green (or was safely identify-only), the fast G0/G1 gate
is named, and the inventory is complete.

> **Refresh trigger:** the instance is itself a derived artifact. When a checkpoint
> notices the gate command changed, a new doc kind appeared, or a convention
> shifted, the instance is stale — re-run this bootstrap. (A checkpoint can include
> "is `instances/<repo>.md` still current?" in its own blast-radius survey.)
