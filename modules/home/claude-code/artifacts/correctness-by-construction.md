# Correctness by construction: building software that cannot be wrong

The most important idea in software engineering is also its most stubbornly underadopted: **programs should be constructed to be correct, not debugged into correctness after the fact.** This thesis, first articulated by Edsger Dijkstra in the early 1970s, has spawned a half-century lineage of formal methods, type systems, specification languages, and engineering disciplines — all converging on the same insight. The practical payoff is real and measurable: the Paris Métro Line 14 driverless system, built using the B-method, has operated for over 25 years with zero safety-critical defects. Amazon credits TLA+ with catching "serious but subtle" bugs in DynamoDB and S3 that survived design reviews, code reviews, and testing. The seL4 verified microkernel has accumulated **zero defects in 15 years** across its 10,000 lines of verified C code.

What makes this moment different from any previous era is that AI is simultaneously creating urgent demand for correctness by construction — AI agents generate code faster than humans can review it — and making formal methods dramatically cheaper through automated proof generation. DeepMind's AlphaProof achieved silver-medal performance at IMO 2024 using Lean 4. DeepSeek-Prover-V2 hits **88.9%** on standard theorem-proving benchmarks. Martin Kleppmann's December 2025 prediction — "AI will make formal verification go mainstream" — captures a genuine inflection point. We are entering an era where specifying what you want may become easier and more reliable than manually coding how to get it.

## From Dijkstra's insight to a mathematical framework for program correctness

The intellectual throughline begins with a philosophical commitment and rapidly becomes mathematical. Dijkstra's famous dictum — "Program testing can be used to show the presence of bugs, but never to show their absence" — first appeared at the 1969 NATO Software Engineering Conference and was elaborated in his *Notes on Structured Programming* (EWD249, 1970). His argument was not merely that testing was insufficient but that the entire practice of write-then-debug was intellectually bankrupt. In *A Discipline of Programming* (1976), he introduced **predicate transformer semantics** — the weakest precondition calculus — giving developers a mathematical tool for deriving programs from specifications rather than discovering correctness through trial and error.

Tony Hoare's "An Axiomatic Basis for Computer Programming" (1969) provided the formal scaffolding. The Hoare triple **{P} S {Q}** — if precondition P holds before executing statement S, postcondition Q holds after — transformed program correctness from an empirical question into a theorem to be proved. Hoare's rules for assignment, composition, conditionals, and loops gave a complete deductive system. His and Dijkstra's approaches are mathematically dual: Hoare logic provides inference rules for verification, while Dijkstra's wp calculus provides algorithms for constructing those proofs. Together they established that correctness is a property of program structure, not of test suites.

The Curry-Howard correspondence deepened this foundation to its logical bedrock. Haskell Curry observed in 1934 that combinator types correspond to logical axioms; William Howard extended this in 1969 to show that proofs in natural deduction correspond exactly to terms in typed lambda calculus. **A well-typed program is literally a proof of its type, viewed as a proposition.** Per Martin-Löf's Intuitionistic Type Theory (1972–1984) extended this to dependent types — types that depend on values — enabling arbitrarily complex specifications to be expressed as types and verified by type-checking. This is the deepest theoretical justification for correctness by construction: programming and proving are the same activity, seen from different angles.

The refinement calculus, originated by Ralph-Johan Back in his 1978 Helsinki PhD thesis, formalized Dijkstra's stepwise refinement as a mathematical lattice. Specifications and programs inhabit the same space; development proceeds by proving refinement relationships that preserve all properties from abstract to concrete. Jean-Raymond Abrial's B-method (formalized in *The B-Book*, 1996) industrialized this idea: systems are modeled as abstract machines in set theory, then refined through proof-carrying steps to executable Ada code. Bertrand Meyer's Design by Contract (Eiffel, 1985–1992) made Hoare triples practical for object-oriented code through preconditions, postconditions, and class invariants embedded directly in source. Leslie Lamport's TLA+ (1994) extended the tradition to concurrent and distributed systems, specifying state transitions as Boolean functions over primed and unprimed variables — simple enough for engineers, powerful enough for AWS.

## Modern techniques that make illegal states unrepresentable

The theoretical lineage matters because it grounds a spectrum of practical techniques, from lightweight type tricks to full formal verification, all sharing the same philosophy: **push correctness guarantees from runtime to compile time, from testing to structure**.

**Type-driven development** — making illegal states unrepresentable — is the most accessible entry point. Yaron Minsky (Jane Street) coined the phrase for OCaml; Scott Wlaschin's *Domain Modeling Made Functional* (2018) popularized it for F#. The core technique uses discriminated unions (sum types) to model domain states so that each variant carries exactly the data relevant to it. A network connection modeled as `Connecting | Connected of {sessionId, lastPing} | Disconnected` eliminates the entire class of bugs where a `lastPingTime` exists while connecting. Phantom types tag values with compile-time metadata (`Html<Sanitized>` vs. `Html<Raw>`). GADTs in Haskell and OCaml refine type parameters per constructor, encoding invariants like "this expression AST is well-typed" directly in the host type system. Even TypeScript's structural type system supports branded types — intersecting a primitive with a unique symbol to create nominal distinctions.

Alexis King's **"Parse, don't validate"** (2019) distilled this philosophy into an architectural principle. Validation checks a property and discards the evidence; parsing checks the same property and returns a type that preserves it. Instead of `validateNonEmpty :: [a] -> IO ()`, prefer `parseNonEmpty :: [a] -> IO (NonEmpty a)`. Internal code then receives `NonEmpty a` and never needs to handle the empty case. This eliminates shotgun parsing — validation scattered throughout a codebase, hoping some check catches every bad input — by stratifying programs into a parsing boundary (where failures happen) and a core (where types guarantee invariants hold). The insight has become a foundational design pattern across Haskell, Rust, TypeScript, and Kotlin ecosystems.

**Refinement types** push type-level guarantees further by annotating types with logical predicates verified by SMT solvers at compile time. Liquid Haskell (Jhala, Vazou, UC San Diego) lets developers write `{v:Int | v > 0}` and have Z3 verify it across all code paths — proving absence of division by zero, buffer overruns, or custom functional properties with roughly **1.7 lines of annotation per 100 lines of code**. F* (Microsoft Research) supports both refinement and full dependent types, and was the vehicle for Project Everest's verified HTTPS/TLS stack. These sit at a practical sweet spot: more expressive than standard types, more automated than full dependent types.

**Effect systems and algebraic effects** make side effects visible in types. Koka (Daan Leijen, MSR) tracks every effect in the function signature — `function divide(x: int, y: int) : exn int` declares that division may raise exceptions. OCaml 5 added algebraic effect handlers for concurrency. Effect-TS brings typed effect tracking to the JavaScript ecosystem, with `Effect<A, E, R>` encoding success type, error type, and required dependencies. When effects are tracked, pure functions cannot accidentally perform I/O, resource lifetimes are explicit, and concurrency primitives become composable library code rather than language primitives.

**Property-based testing** bridges the gap between testing and proving. QuickCheck (Claessen and Hughes, 1999) generates random inputs to falsify universal properties, then shrinks failures to minimal counterexamples. It is not proof, but it forces developers to articulate invariants — "for all inputs, encoding then decoding yields the original value" — and systematically explores edge cases humans miss. Hypothesis (Python), fast-check (TypeScript), and QuviQ QuickCheck (Erlang, commercial) have found bugs in production systems that passed extensive unit test suites.

At the heavyweight end, **Ada/SPARK** represents perhaps the purest practical embodiment of correctness by construction. SPARK is a formally verifiable subset of Ada where GNATprove mathematically proves absence of runtime errors — no buffer overflows, no null dereferences, no integer overflows — for all possible inputs, not just tested ones. The Praxis Critical Systems methodology, documented by Anthony Hall and Roderick Chapman in their 2002 IEEE Software paper "Correctness by Construction: Developing a Commercial Secure System," used SPARK to achieve defect rates dramatically lower than CMM Level 5 organizations. SPARK is used on the Eurofighter Typhoon, the UK's iFACTS air traffic management system, Hitachi Rail STS signaling (SIL 4), and medical devices including left ventricular assist devices.

## Industrial proof points: where correctness by construction has worked at scale

The strongest argument for correctness by construction is not theoretical but empirical. Several large-scale deployments provide hard evidence.

The **Paris Métro Line 14** project remains the landmark case. Inaugurated October 15, 1998, it was the first high-capacity driverless metro in a capital city. The safety-critical control software was developed entirely using Abrial's B-method: **115,000 lines of B models** generated 86,000 lines of Ada code, with **27,800 proof obligations** discharged (92% automatically). No unit testing was performed — formal development and proof gave sufficient confidence. During an eight-month trial period, the system controlled approximately 96,000 trains with no fault observed. Over 25 years later, Line 14 is the busiest line of the Paris Métro, carrying **~820,000 riders per weekday**, and the B-method has been adopted for subsequent systems including Line 1 automation (COPPILOT) and OCTYS.

**Amazon's adoption of TLA+**, documented by Newcombe et al. in their 2014/2015 CACM paper, is the most influential industry case study for distributed systems. Seven AWS teams used TLA+ to specify DynamoDB, S3, and other critical infrastructure. The model checker found serious bugs that had survived extensive code review and testing — including bugs in fault-tolerant replication that would have been extraordinarily difficult to reproduce. One model evaluation explored **31 billion states** over five weeks on an EC2 instance. The cultural impact was as significant as the technical results: executive management proactively encouraged teams to write TLA+ specifications, and Chris Newcombe called it "the most valuable thing that I've learned in my professional career."

**Meta's Infer static analyzer**, built on Peter O'Hearn's separation logic, runs on every code modification across Facebook, Instagram, WhatsApp, and Messenger — codebases of millions of lines with 1,000+ daily changes. By 2015, Infer was catching over **1,000 bugs per month** before production, with a 70%+ fix rate at diff time. RacerD, its concurrency analyzer, enabled Facebook's conversion of its Android News Feed from single-threaded to multithreaded architecture; a Facebook engineer stated that "without Infer, multithreading in News Feed would not have been tenable." External adopters include Amazon, Mozilla, Spotify, and Uber.

The **seL4 verified microkernel** (NICTA/Data61, led by Gerwin Klein and Gernot Heiser) is the gold standard for full functional correctness verification. Its **8,700 lines of C** have a complete Isabelle/HOL proof that the implementation satisfies its abstract specification — covering absence of buffer overflows, code injection, null dereferences, and memory leaks with mathematical certainty. The total effort was ~20 person-years, but the kernel has accumulated **zero defects over 15 years of deployment** in defense, aerospace, and autonomous vehicle applications. The verified CompCert C compiler (Xavier Leroy, INRIA) achieved similar results: random testing with Csmith devoted six CPU-years to finding miscompilation bugs and found none, while every other tested compiler had middle-end bugs. In March 2026, CompCert achieved a milestone — the first DO-178C/DO-333/DO-330 certification credits claimed from compiler usage, for the ATR 42/72 aircraft.

The **economic case** is strong. A 2002 NIST study estimated software bugs cost the U.S. economy **$59.5 billion annually**; by 2016, CISQ put the figure at $1.1 trillion. Barry Boehm's classic finding — defects found after delivery cost **up to 100× more** than those caught during design — provides the theoretical basis for "shifting left." Anthony Hall's analysis of Correctness by Construction projects found them "demonstrably cost-effective" with dramatically lower defect rates. AWS engineers reported that writing TLA+ specifications was "both more reliable, and also less time consuming" than informal proofs. The pattern across industries is consistent: high upfront investment in specification and proof, followed by radically lower costs during integration, testing, and operations.

## AI makes correctness by construction both more urgent and more tractable

The most consequential development is the collision of two trends: AI generating code faster than humans can verify it, and AI making formal verification dramatically cheaper.

On the urgency side, the data is sobering. Stack Overflow's 2025 survey shows **65% of developers** use AI coding tools weekly. CodeRabbit's January 2026 analysis found AI-generated code introduces security bugs at **1.5–2× the rate** of human coders, excessive I/O operations at ~8× higher rates, and concurrency errors at 2× the rate. Anthropic's 2026 Agentic Coding Trends Report notes that engineers keep tasks for themselves when "correctness matters more than convenience." The implication is clear: when AI writes code at scale, human review becomes the bottleneck, and **machine-verifiable constraints become the only scalable alternative**.

On the tractability side, progress in AI-assisted theorem proving has been explosive. DeepMind's **AlphaProof** (July 2024, published in Nature November 2025) achieved silver-medal performance at IMO 2024, solving 3 of 6 problems including the hardest (solved by only 5 of 609 human participants) using reinforcement learning combined with Lean 4. **DeepSeek-Prover-V2** (April 2025), an open-source 671B-parameter model, achieves 88.9% on the MiniF2F benchmark and solved 49 PutnamBench problems. **Goedel-Prover-V2** (August 2025) achieves 90.4% on MiniF2F at 32B scale with self-correction. Multiple systems achieved gold-medal performance at IMO 2025. Galois reported that Claude Code is "pretty good at interactive theorem proving," pointing toward "a future where theorem proving is solved — cheap, abundant, and automatic."

The emerging paradigm is **"vericoding"** — a term coined by Bursuc, Tegmark et al. (September 2025, accepted at POPL 2026): LLM generation of formally verified code from formal specifications, as opposed to unverified "vibe coding." Their benchmark of 12,504 formal specifications found an **82% success rate in Dafny** using off-the-shelf LLMs, with LLM progress improving pure Dafny verification from 68% to 96% over a single year. Martin Kleppmann's economic argument is compelling: seL4 required 20 person-years and 200,000 lines of Isabelle proof for 8,700 lines of C — 23 lines of proof per line of implementation. AI-generated proofs could collapse this ratio by orders of magnitude. AutoVerus (OOPSLA 2025) already demonstrates >90% automated proof generation for Rust code via Verus, with >50% of tasks solved in under 30 seconds.

The convergence of type systems, effect systems, and formal verification is creating a practical sweet spot. Rust's ownership and borrowing system provides memory safety and data-race freedom as compile-time guarantees — properties that historically required formal verification. **RefinedRust** (PLDI 2024) layers refinement types on top, enabling full functional correctness proofs for real Rust code. **Verus** (SOSP 2024, winning two of three best paper awards) is seeing industrial adoption at Microsoft and Amazon for verified systems code. Mainstream languages continue moving toward more expressive type systems — TypeScript's type-level computation, Kotlin's contracts, even the C++26 contracts proposal (P2900). The spectrum from simple types through rich types (Rust ownership) to refinement types (Liquid Haskell) to dependent types (Lean 4, Idris 2) is becoming more continuous, with each step requiring less additional effort.

Dependent types themselves are becoming more practical. **Lean 4**, developed by Leonardo de Moura, has emerged as the dominant platform for convergence of programming and proving — used by essentially all recent AI+formal methods work, by Terence Tao's collaborative proof of the Polynomial Freiman-Ruzsa conjecture, and supported by the extensive Mathlib library. **Idris 2** (Edwin Brady, 2021) introduced Quantitative Type Theory, unifying linear and dependent types. Full-spectrum dependent typing remains primarily a research tool, but the consensus is shifting: AI may close the gap by generating the proofs that make dependent types usable by non-experts.

## Conclusion: the specification becomes the program

The intellectual throughline from Dijkstra to vericoding is remarkably coherent. The core thesis has never changed: correctness is a property of construction, not of post-hoc verification. What has changed is the practicality. Dijkstra's insight was philosophical; Hoare made it mathematical; Abrial made it industrial; modern type systems made it accessible to everyday programmers; and AI is making the most powerful techniques — full formal verification — cheap enough for mainstream adoption.

Three novel insights emerge from this analysis. First, **AI does not replace correctness by construction — it demands it**. The faster code is generated, the less viable human review becomes as a quality gate, and the more essential machine-checkable specifications become. Second, **the distinction between specification and implementation is collapsing**. Vericoding treats the formal specification as the primary artifact; the implementation and its proof are generated. This fulfills Dijkstra's original vision more completely than he likely imagined — the programmer specifies what, and the machine constructs a provably correct how. Third, **the economic barrier to formal methods is falling faster than the cultural barrier**. The tools are ready; the challenge is convincing engineering organizations that formal specification is not an academic luxury but a practical necessity — especially in a world where AI agents will write most code and humans must still trust the systems those agents build.
